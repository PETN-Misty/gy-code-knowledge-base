### 4. RAG 流程说明（5 句话）

> 1. **文档导入**：用户上传 `.txt` / `.md` 文件或通过 API 提交文本，后端按段落和句子自动切分为约 500 字的块，存入 `knowledge_chunks` 表。
> 2. **向量索引构建**：启动时或文档变更后，系统从数据库取出所有知识块文本，提取字符 1~4-gram 特征，用 TF-IDF 算法为每个块计算稀疏向量，加载到内存中供快速检索。
> 3. **语义检索**：用户提问时，系统对查询文本做同样的 n-gram 向量化，与所有知识块向量计算余弦相似度，按相关度降序返回 topK 个最匹配的块。
> 4. **上下文组装**：将命中的知识块文本按 `【来源：xxx】正文内容` 格式拼接成上下文，传入 DeepSeek 聊天 API。
> 5. **生成回答**：DeepSeek 基于提供的参考资料生成带来源引用的中文回答，返回给用户。

### 3. topK 改变后的影响

| topK 值 | 命中块数 | 特点 |
|---------|---------|------|
| **topK=2** | 2 块 | 回答更聚焦，但可能遗漏辅助信息 |
| **topK=5**（默认） | 5 块 | 平衡相关性与上下文丰富度 |
| **topK=8** | 8 块 | 上下文更丰富，但可能引入噪声 |

**具体影响：**
- **topK 越小** → LLM 看到的参考越少，回答越精确但不全面，适合问题有明确单一答案的场景
- **topK 越大** → LLM 看到的参考越多，回答可能更全面，但低相关度的块可能分散注意力，增加 Token 消耗，且可能引入矛盾信息
- **推荐值 3~7**：根据知识库规模和问题复杂度动态调整

## 技术架构总结

```
┌──────────┐    ┌──────────────┐    ┌───────────┐    ┌──────────┐
│ 用户提问  │───▶│ N-gram 向量化 │───▶│ 余弦相似度  │───▶│ TopK 排序 │
└──────────┘    └──────────────┘    └───────────┘    └──────────┘
                                                          │
                    ┌──────────────────────────────────────┘
                    ▼
            ┌──────────────┐    ┌──────────┐    ┌──────────┐
            │ 拼装 Context  │───▶│ DeepSeek  │───▶│ 返回回答  │
            └──────────────┘    └──────────┘    └──────────┘
```

- **向量引擎**：纯本地 n-gram TF-IDF（字符 1~4-gram，零外部依赖）
- **检索方式**：全量加载 + 余弦相似度（适配小规模知识库）
- **生成模型**：DeepSeek V4 Flash（`deepseek-chat`）
- **存储**：MySQL 8.0，`knowledge_chunks` 表
- **切片策略**：按段落/句子切分，~500 字/块，50 字重叠
